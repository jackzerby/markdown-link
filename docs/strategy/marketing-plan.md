# Marketing Plan

## Positioning

`markdown.link` is the fastest way for founders and technical teams to turn a
markdown file into a shareable link.

The team should keep an internal research loop on adjacent agent-native
products and founder accounts so messaging stays grounded in what the market is
already responding to.

The short promise:

- write in markdown
- run one command
- get a clean link

The sharper product wedge:

- terminal-native sharing for people already using Codex and Claude Code
- a better way to ship plans, memos, specs, and launch notes
- a link format that is cleaner than pasting a file or screenshot

## Target segments

### Primary

- startup founders
- indie hackers
- technical operators
- engineers who use Codex and Claude Code every day
- founders who keep everything in markdown and ship from the terminal

### Secondary

- growth engineers
- DevRel teams
- AI-native agencies
- product managers who share specs in markdown
- technical founders who post workflows on X

## Messaging pillars

### 1. Speed

The file already exists. Publishing should take seconds, not a new workflow.

### 2. Agent-native

Codex or Claude Code can generate the markdown and publish it without leaving
the terminal.

### 3. Shareability

Turn local markdown into a readable public artifact.

### 4. Control

Claim links, keep them live, add passwords, and use your domain.

## Offer Ladder

### Free

- one-click anonymous publish
- short-lived links for lightweight sharing
- claim flow for important content

### Paid

- permanent links
- custom domains
- stronger control and team workflows
- fewer expiry surprises

## Launch plan

### Week 1: founder-led demos

- post one short CLI demo per day on X
- publish examples for investor updates, launch plans, team memos, and specs
- reply to builders already sharing screenshots of markdown, Codex, and Claude Code
- reuse the same demo structure until it starts converting
- study adjacent product onboarding, CTA structure, and founder posting rhythm
  before drafting new variants

### Week 2: proof of fit

- show real before/after workflows
- show anonymous publish, claim, and permanent link flows
- show one paid-plan unlock per post
- publish a short story around a real markdown file being shared

### Week 3: repetition and variation

- repost the best-performing demo in a different framing
- ship one visible improvement every few days and post it
- turn user examples into social proof
- keep the loop tight: build, post, measure, repeat

## Content system

### Post types

- one-command demos
- before/after screenshots
- founder memos published from terminal
- Codex and Claude Code workflow clips
- "share this plan in 10 seconds" examples
- comparisons against ugly alternatives

### Reusable hook templates

- "I stopped pasting markdown into docs and started doing this instead."
- "If your agent can write it, it should be able to publish it."
- "Founders should be able to share plans from the terminal."
- "This is the cleanest way I’ve found to send markdown to a teammate."

## Distribution loops

- demo on X
- click through to landing page
- publish a first file
- claim the link if it matters
- hit expiry pain or need control
- upgrade

## Organic motion on X

### Where to show up

- founder posts about launch notes, specs, and operating docs
- Codex and Claude Code discussion threads
- AI tooling replies where people are already sharing workflows
- product-building conversations where markdown is being used manually
- adjacent founder accounts that visibly convert product work into repeatable
  public demos

### Reply strategy

- monitor people talking about markdown, specs, launch notes, Codex, and Claude Code
- reply with a working example instead of a pitch
- keep replies specific and useful
- prefer one concrete command over a long explanation

### Weekly cadence

- 3 original demos
- 10 to 20 targeted replies
- 1 founder story
- 1 feature or workflow update

## Paid motion on X

Start only after organic proof exists.

- use website tagging and retargeting on site visitors
- retarget users who hit pricing, claim, and install pages
- promote the best-performing organic demos rather than inventing ad creative from scratch
- only spend once the CLI and first-publish funnel are producing repeatable conversions

## Conversion funnel

1. see a demo on X
2. visit landing page
3. install CLI or publish one file
4. share first link
5. hit expiry pain or need control
6. upgrade

## Metrics

- visits from X
- install rate
- first publish rate
- first share rate
- anonymous claim rate
- free-to-paid conversion
- reply engagement
- demo-to-install conversion
- paid-plan uplift from claim/expiry pain

## What not to do

- do not market this like a generic docs app
- do not lead with features before the command
- do not overproduce content that does not show a real workflow
- do not chase broad audiences before the founder/operator wedge is working
- do not copy another product's public copy line-for-line; learn the mechanic
  and adapt it to markdown.link
