# Team Hiring Plan

## Purpose

This is the operating team for `markdown.link` as a founder-led, agent-powered
startup.

The goal is not a big org chart. It is a small team with clear ownership that
can ship product and distribution every week.

## Hiring order

### 1. CEO

Owns:

- priorities
- tradeoffs
- what the company says yes to
- what the company says no to

### 2. Product strategy

Owns:

- roadmap clarity
- user pain synthesis
- turning support and feedback into buildable issues

### 3. Growth lead

Owns:

- X distribution
- launch planning
- reply strategy
- converting features into shareable demos

### 4. Community/reply agent

Owns:

- monitoring relevant conversations
- drafting replies
- finding posts that can be turned into examples

### 5. Analytics/ops agent

Owns:

- weekly scorecards
- top-of-funnel metrics
- publish/claim/conversion tracking
- failure detection

### 6. Business/pricing agent

Owns:

- pricing recommendations
- packaging clarity
- upgrade friction analysis

### 7. Engineering swarm

Owns:

- product delivery
- bug fixes
- reliability work
- follow-up changes from launch feedback

## What each role produces

### CEO outputs

- weekly memo
- prioritized issue list
- launch/no-launch call

### Product outputs

- roadmap note
- issue breakdowns
- feature framing for the team

### Growth outputs

- post drafts
- reply drafts
- launch scripts
- social proof ideas

### Community outputs

- conversation shortlist
- reply queue
- opportunity notes

### Analytics outputs

- daily metric summary
- weekly GTM summary
- conversion notes

### Engineering outputs

- merged changes
- verified builds
- short release notes

## Weekly cadence

1. Monday: CEO sets priorities.
2. Tuesday to Thursday: product, growth, and engineering execute.
3. Friday: analytics and CEO review what converted.
4. Weekend: keep one lightweight growth agent running for reply opportunities.

## How the team should work

- keep each issue scoped narrowly
- write outcomes in markdown
- use CLI-first workflows whenever possible
- prefer one owner per issue
- keep review loops short

## Founder rule

If a task cannot be explained in one short sentence, it is probably too broad
for the current week.
